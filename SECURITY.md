# 安全維護規範

本專案後續可能會使用 LINE、簡訊、Email、資料庫或第三方服務 API。所有 API KEY、Token、密碼與私鑰都必須視為機密資料處理。

## 核心原則

- 不要把任何真實 API KEY、Token、密碼、私鑰寫進程式碼。
- 不要把真實機密資訊寫進 README、文件、簡報、測試資料或截圖。
- 不要提交 `.env`、憑證 JSON、私鑰檔、服務帳號檔。
- 只提交 `.env.example`，用空值或假資料示範需要哪些設定。
- 本機開發使用 `.env`，正式環境使用部署平台的 Secret / Environment Variables。
- 若金鑰曾經被提交、貼出或截圖外流，應立即停用並重新產生。

## 建議做法

1. 本機建立 `.env`

   從 `.env.example` 複製一份 `.env`，再把真實金鑰填入 `.env`。

2. 程式讀取環境變數

   程式應從環境變數讀取設定，例如 `LINE_CHANNEL_ACCESS_TOKEN`、`SMS_API_KEY`、`DATABASE_URL`。

3. 錯誤訊息遮蔽

   前端畫面、後台、log 與錯誤訊息不應完整顯示金鑰。若需要顯示，只能顯示前後少數字元，例如 `sk-...abcd`。

4. 權限最小化

   第三方服務的金鑰只開必要權限，不共用管理者主帳號密碼。

5. 定期檢查

   每次提交前檢查是否誤放 `.env`、憑證 JSON、私鑰或真實 Token。

## 本專案已採取的防護

- `.gitignore` 已排除 `.env`、私鑰、常見憑證 JSON 與 secrets 資料夾。
- `.env.example` 只列出欄位，不包含真實值。
- 後續開發時，任何 API KEY 都應透過環境變數或部署平台 Secret 注入。

## 未來開發提醒

若功能需要串接 LINE、簡訊、Email、付款、資料庫或監理相關服務，開發前先確認：

- 金鑰放在哪裡？
- 誰有權限存取？
- 是否會出現在前端原始碼？
- 是否會被寫入 log？
- 是否需要區分測試環境與正式環境？
